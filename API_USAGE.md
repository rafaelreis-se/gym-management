## üîê Authentication Flow

### Scenario 1: Admin Creates Student ‚Üí Student First Access

#### Step 1: Admin creates student (no password)
```bash
POST /api/auth/register
{
  "email": "maria@example.com",
  "role": "STUDENT",
  "studentId": "student-uuid-here"
  // No password = first access required
}

Response:
{
  "id": "user-uuid",
  "email": "maria@example.com",
  "isFirstAccess": true,  ‚Üê User needs to set password
  ...
}
```

#### Step 2: Send email to student with link
```
Email: "Welcome! Set your password: http://app.com/first-access?userId=user-uuid"
```

#### Step 3: Student sets password
```bash
POST /api/auth/set-password
{
  "userId": "user-uuid",
  "password": "SecurePass123!"
}

Response:
{
  "message": "Password set successfully"
}
```

#### Step 4: Student logs in
```bash
POST /api/auth/login
{
  "email": "maria@example.com",
  "password": "SecurePass123!"
}

Response:
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "a1b2c3d4e5f6...",
  "user": {
    "id": "user-uuid",
    "email": "maria@example.com",
    "role": "STUDENT",
    "studentId": "student-uuid-here",
    ...
  }
}
```

## üèÉ Common Workflows

### New Student Registration (by Admin)
```bash
1. POST /api/students ‚Üí Create student
2. POST /api/guardians ‚Üí Create guardian (if child)
3. POST /api/guardians/link-to-student ‚Üí Link guardian to student
4. POST /api/auth/register ‚Üí Create user account for student/guardian
5. POST /api/enrollments ‚Üí Create enrollment
6. Send welcome email with first access link
```

### Payment Flow
```bash
1. GET /api/financial/plans/active ‚Üí List available plans
2. POST /api/enrollments ‚Üí Enroll student
3. System auto-generates payment schedule
4. GET /api/financial/payments/pending ‚Üí Check pending payments
5. PATCH /api/financial/payments/:id/mark-as-paid ‚Üí Mark as paid
6. GET /api/financial/payments/overdue ‚Üí Monitor overdue
```

### Graduation Flow
```bash
1. Instructor decides to promote student
2. POST /api/graduations ‚Üí Record graduation
   {
     "studentId": "...",
     "modality": "JIU_JITSU",
     "beltColor": "BLUE",
     "beltDegree": "DEGREE_1",
     "graduationDate": "2025-10-15",
     "grantedBy": "Professor Jo√£o"
   }
3. Student/Guardian can see via GET /api/graduations/student/:id
```

### Product Sale
```bash
1. GET /api/products/active ‚Üí List available products
2. POST /api/products/sales ‚Üí Create sale
   {
     "studentId": "student-id",
     "items": [
       { "productId": "kimono-id", "quantity": 1 },
       { "productId": "belt-id", "quantity": 1 }
     ],
     "paymentMethod": "CREDIT_CARD",
     "discountAmount": 10.00
   }
3. System auto-updates stock
4. GET /api/products/low-stock ‚Üí Check if reorder needed
```

## üìä Database Relationships

```

  ‚îú‚îÄ id (PK)

  ‚îú‚îÄ id (PK)

  ‚îú‚îÄ id (PK)

  ‚îú‚îÄ relationship (FATHER, MOTHER, etc.)

  ‚îú‚îÄ id (PK)
  ‚îú‚îÄ email (unique)

  ‚îú‚îÄ role (ADMIN, STUDENT, GUARDIAN, INSTRUCTOR)
```
